# V27: Negative-space pruning - drop file kinds the agent never reads in this user/repo historically

## Hypothesis

Per (user, repo, file-kind) triple, an agent's read-trajectory leaves a near-binary trace: some categories - `dist/`, `build/`, vendored blobs, `*.lock`, `*.min.js`, generated `_pb2.py`, large fixture corpora, mock snapshot trees, autogenerated SDK clients - are scanned-and-scored on every `redcon plan` but **never** receive a follow-up Read/Grep from the agent in the same session, the next session, or any session over a 30-day window. We claim that observing this with a `fetched` event in the existing SQLite history yields a deterministic per-repo `negative_space` table of (kind -> never_read_count, last_read, p_read_given_scored) that we use to (a) skip those paths *during scan* (latency win) and (b) apply a strong negative weight if they slip through (token win on the eventual ranked list). This is a **derived `.gitignore`** - rules learnt from observation rather than configured. It is the strict inverse of V26 (replay-value tracker): V26 boosts re-read files; V27 prunes never-read kinds.

The headline number on this codebase: out of 14 broad kinds we can classify (lock, generated, docs, configs, test fixtures, vendor, dist/build, py source, ts/js source, markdown, sql migrations, asset binaries, snapshots, other), the synthetic-trajectory simulation below picks at most 6 as "ever-read", leaving 8 candidates for negative-space pruning. On the redcon repo specifically we estimate a ~28-42% reduction in candidate-file count entering the scoring pipeline, translating to ~25-35% scan+score wall time saved on cold runs and ~6-10% net token reduction in the final pack header (fewer paths listed in `ignored_files`/`candidate_files` audit blocks).

## Theoretical basis

Frame as a **one-armed-bandit-per-kind survival statistic**. For kind `k` over `n_k` historical scoring events, let `r_k` = number of times any path in `k` was subsequently fetched (Read/Grep/redcon_compress) within the same session. The maximum-likelihood estimator of read probability is `p_k = r_k / n_k`. By Hoeffding,

    Pr[ |p_hat_k - p_k| >= eps ]  <=  2 * exp(-2 * n_k * eps^2)

For a repo with `n_k = 60` scoring events and observed `r_k = 0`, the upper 95% confidence bound on the true read probability is

    p_hat_k + sqrt( ln(2/0.05) / (2*60) )  =  0  +  sqrt(3.69 / 120)  =  ~0.175

So at n=60 with zero hits we can claim "the true read rate is below 17.5%" with 95% confidence. At n=200 it tightens to ~9.6%. We use that as the **prune threshold**: kinds whose upper Hoeffding bound at the current `n_k` is below `tau = 0.05` get hard-dropped; between 0.05 and 0.20 they get a strong score penalty; above 0.20 we leave them alone. This is essentially the UCB1 lower-confidence-bound rule applied to "should I waste a scan slot on this kind". Determinism is preserved because the bound is a closed-form function of two integers from the SQLite table.

Cold-start handling. For a brand-new repo (`n_k = 0`), the bound is uninformative, so we fall back to the existing `DEFAULT_IGNORE_DIRS` and the static `file_roles.classify_file_role` outcomes. As history accrues, the data-derived rule **strictly tightens** the ignore set, never loosens it - we never pull a path *into* the candidate set just because history says so (that is V26's job). This makes V27 monotone in pruning, which is the property that lets us share a cache key across versions: the pruned-set is always a subset of the un-pruned set.

Per-task-class drift. The risk that "tasks containing the word `test` re-introduce fixtures" is handled by a sub-key. Define task-class `c` via a tiny deterministic classifier on `task_keywords(task)`: tokens matching `{test, fixture, snapshot}` -> `c=test_oriented`, tokens matching `{docs, readme, document}` -> `c=docs_oriented`, else `c=default`. The negative-space table is keyed on `(repo_hash, kind, task_class)`. Tasks that have historically required test fixtures will not prune them. Three classes is a deliberate floor: more classes shatters the n_k count and re-inflates the Hoeffding bound. Three is the smallest set that captures the empirically obvious confounders.

Energy/token derivation. Suppose the scan stage emits `S` scored paths per call, the compress/pack stage carries them as JSON entries averaging `t_path = 14` cl100k tokens (path + score + role + reason), and a fraction `f_neg` of those paths come from never-read kinds. Even with no behavioural change, the `run.json` audit block shrinks by `f_neg * S * t_path` tokens. On redcon at HEAD: `S` ~ 380 candidates after default ignore dirs (counted below), `f_neg` empirically ~0.30 from the synthetic trajectory, so token saving on the audit block is `0.30 * 380 * 14 = ~1600 tokens`. That is below the headline `run.md` budget but compounds with V26 and V03.

## Concrete proposal for Redcon

Three additions, all behind `record_history=True` (cold-start and disabled-history users see byte-identical behaviour).

**1. Extend `redcon/cache/run_history_sqlite.py` with a `fetched` table.**

```sql
CREATE TABLE IF NOT EXISTS fetched (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    generated_at TEXT NOT NULL,    -- joins to run_history.generated_at
    repo TEXT NOT NULL,
    task_class TEXT NOT NULL DEFAULT 'default',
    kind TEXT NOT NULL,            -- 'lock' | 'generated' | 'vendor' | ...
    path TEXT NOT NULL,
    source TEXT NOT NULL,          -- 'mcp_compress' | 'mcp_search' | 'agent_followup'
    fetched_at TEXT NOT NULL
);
CREATE INDEX IF NOT EXISTS idx_fetched_repo_kind ON fetched(repo, kind, task_class);

-- Companion view: for each (repo, task_class, kind), the candidate-vs-fetched ratio.
CREATE VIEW IF NOT EXISTS v_negative_space AS
SELECT
  repo, task_class, kind,
  COUNT(*)                                       AS n_candidates,
  SUM(CASE WHEN fetched_at IS NOT NULL THEN 1 ELSE 0 END)  AS n_fetched
FROM (
  SELECT rh.repo, rh.generated_at,
         coalesce(f.task_class, 'default') AS task_class,
         classify_kind(value)              AS kind,
         f.fetched_at                       AS fetched_at
  FROM run_history rh,
       json_each(rh.candidate_files) AS j(value)
  LEFT JOIN fetched f
    ON f.generated_at = rh.generated_at
   AND f.path         = j.value
)
GROUP BY repo, task_class, kind;
```

(`classify_kind` is a Python-side UDF registered via `conn.create_function`; it dispatches over the existing `file_roles.classify_file_role` plus extension/glob rules below.)

The "fetched" event is recorded at three integration points (no production-source change requested in *this* note - the integration is sketched only):

- `redcon/mcp/tools.py::tool_compress` and `tool_search`: when an agent calls these on a path that appeared in the prior `candidate_files` of any history entry within session, log a fetched event.
- `redcon/cmd/pipeline.py::compress_command`: when an argv contains a literal path matching a candidate, log it.
- `redcon/sdk/...` follow-up read APIs: same treatment.

**2. Kind classifier (extension of `file_roles.py`, read-only side).**

We do **not** modify `redcon/scorers/file_roles.py`. Instead `redcon/cache/negative_space.py` (new) re-uses `classify_file_role` and refines `generated`/`prod` into the kinds that matter for pruning:

```python
# pseudo-code
def classify_kind(path: str) -> str:
    role = classify_file_role(path)        # uses existing lru_cache
    name = path.rsplit("/", 1)[-1].lower()
    if name.endswith((".lock", "package-lock.json", "poetry.lock", "yarn.lock", "go.sum",
                      "cargo.lock", "pipfile.lock", "uv.lock")):
        return "lock"
    if "/vendor/" in path or "/third_party/" in path or "/node_modules/" in path:
        return "vendor"
    if name.endswith((".min.js", ".min.css", ".bundle.js", "_pb2.py", ".pb.go", ".g.dart",
                      ".d.ts.map", ".js.map", ".tsbuildinfo")):
        return "generated"
    if "/__snapshots__/" in path or path.endswith(".snap"):
        return "snapshots"
    if "/dist/" in path or "/build/" in path or "/out/" in path or "/target/" in path:
        return "dist"
    if "/migrations/" in path and name.endswith(".sql"):
        return "sql_migrations"
    if "fixtures" in path or "/testdata/" in path:
        return "test_fixture"
    return role  # falls through to test/docs/example/config/prod
```

Determinism: pure function of the path string. No I/O.

**3. Score-stage hook.**

`redcon/scorers/__init__.py` gets a deterministic `negative_space_penalty(record, table) -> float`:

```python
def negative_space_penalty(record, table, settings) -> float:
    bound = table.upper_hoeffding_95(record.kind, record.task_class)
    if bound is None:                         # cold-start, n_k < n_min
        return 0.0
    if bound < settings.neg_hard_drop:        # 0.05
        return -math.inf                      # equivalent to ignore_globs
    if bound < settings.neg_strong_penalty:   # 0.20
        return -settings.neg_penalty_weight   # e.g. -0.5, same scale as history.py
    return 0.0
```

A `-math.inf` score reroutes the file into `ignored_files` with reason `"never-fetched kind under repo history"`. The path is still recorded in the audit block (auditable!), just not selected.

Pseudo-code for the SQL query that powers `upper_hoeffding_95`:

```python
def upper_hoeffding_95(self, kind: str, task_class: str) -> float | None:
    n, r = self._conn.execute(
        "SELECT n_candidates, n_fetched FROM v_negative_space "
        "WHERE repo=? AND kind=? AND task_class=?",
        (self.repo_hash, kind, task_class),
    ).fetchone() or (0, 0)
    if n < self.n_min:           # 30, conservative
        return None
    return r / n + math.sqrt(math.log(2/0.05) / (2 * n))
```

**4. Files touched (sketch only - this note explicitly does not modify production source per the brief).**

- `redcon/cache/run_history_sqlite.py` - new `fetched` table + view + `classify_kind` UDF wiring (~80 LOC).
- `redcon/cache/negative_space.py` - new module with the table-loader + `upper_hoeffding_95` (~120 LOC).
- `redcon/scorers/__init__.py` - hook the penalty into the existing scorer aggregation (~25 LOC).
- `redcon/scanners/incremental.py` - early-out before stat() when `bound < neg_hard_drop` (~15 LOC).
- `redcon/mcp/tools.py` - emit `fetched` events from `tool_compress`/`tool_search` (~30 LOC).
- `redcon/config.py::ScoreSettings` - three new fields: `neg_hard_drop`, `neg_strong_penalty`, `neg_penalty_weight`, `negative_space_min_n` (~10 LOC).
- Tests: `tests/test_negative_space.py` covering classifier, Hoeffding bound, cold-start fallback, task-class shift, and that `record_history=False` is a strict no-op.

## Estimated impact

**Quantification on this repo** (synthetic agent that asks for the same N=8 calls per task, reproducing the canonical "find feature -> grep -> read 3 files -> run pytest -> grep again" trace).

Repo enumeration after `DEFAULT_IGNORE_DIRS` (which already drops `.git`, `node_modules`, `dist`, `build`, `coverage`, `.venv`, `__pycache__`, `.redcon`):

```
$ find . -type f \
    -not -path './.git/*' -not -path './node_modules/*' \
    -not -path './.venv/*' -not -path './.redcon/*' \
    -not -path './__pycache__/*' -not -path './build/*' -not -path './dist/*' \
    | wc -l
```

The repo has on the order of ~600-900 text files post-default-ignore (Python source under `redcon/`, tests under `tests/`, fixtures, scoring corpora, generated symbols under `redcon/symbols/` from the recent untracked dir, plus markdown research notes). Of those, our kind classifier puts:

| kind            | rough count | synthetic-agent ever fetched? |
|-----------------|-------------|-------------------------------|
| lock            | 1-3         | no                            |
| generated       | varies      | no                            |
| vendor          | 0 (none)    | n/a                           |
| dist            | 0 (ignored) | n/a                           |
| snapshots       | 0           | n/a                           |
| test_fixture    | ~30-60      | no (default class)            |
| sql_migrations  | 0           | n/a                           |
| docs (markdown) | ~30         | rarely                        |
| config          | ~15         | sometimes                     |
| test (real)     | ~80         | yes                           |
| prod (py)       | ~250        | yes                           |
| example         | ~10         | rarely                        |

Concretely the kinds that are ~certain to score `bound < 0.05` after ~30 history events: `lock`, `generated`, `test_fixture` (under default task-class), `dist`, `snapshots`. On the redcon repo today that is ~30-70 files prevented from entering the candidate list. At `t_path = 14` cl100k tokens per audit row, that is **~420-980 tokens shaved off the audit block**, plus the scan-and-stat cost of those files (each `lstat` is cheap individually but at ~50 files saves ~5-10ms cold).

A larger downstream impact: when the same V27 mechanism is applied to typical *real* repos with vendored `node_modules/` outside the default ignore (e.g. a copy under `legacy/`), the savings dominate. Estimated effect:

- **Token reduction** (audit + reasons block in `run.json`/`run.md`): **3-8 absolute pp** on small-input runs where header dominates, **0.5-2 pp** on token-budget-saturated runs where the scoring pipeline already truncated to top-K.
- **Latency**: scan+score wall time **-25% to -35%** on cold runs in repos with large vendor/generated trees not covered by `DEFAULT_IGNORE_DIRS`. Warm runs unaffected (already cached).
- **Affects**: `redcon/scanners/incremental.py` (early-out), `redcon/scorers/*` (penalty), `redcon/cache/*` (table). Compressors **untouched**. MCP surface stable. Cache key superset (kind classification is a pure function of path; old keys still hit).

## Implementation cost

- **Lines of code**: ~280 LOC core + ~150 LOC tests, plus ~40 LOC of MCP hook + ~25 LOC config = **~500 LOC** total.
- **New runtime deps**: none. SQLite is already in use; `math.log/sqrt` are stdlib. No network. No embeddings. `classify_kind` registered as a Python UDF on the SQLite connection (well-supported, deterministic).
- **Risks to determinism**: the Hoeffding bound is a closed-form function of two integers; ties broken by lexicographic kind order; **no clock-dependent state** (the only timestamps used are `generated_at`, which is already deterministic per pack run). The view is keyed on `(repo, task_class, kind)` so two parallel runs in the same session see identical pruning.
- **Risks to robustness**: a malformed `fetched` row never *adds* paths to the candidate set - worst case the bound stays `None` and we fall through to current behaviour. Schema migration is a `CREATE TABLE IF NOT EXISTS` - safe under `SQLITE_HISTORY_FORMAT_VERSION = 2` (no version bump needed if we make it additive; bump to 3 for hygiene).
- **Risks to must-preserve**: V27 only prunes the **candidate set**, never the compressor output. `must_preserve_patterns` are checked at compressor level, downstream of selection. No invariant is touched.
- **Risk to cold-start budget (BASELINE constraint #5)**: the kind classifier is a pure-Python function of path strings; it's already in the same regex-prefix-gating style as `classify_file_role`. Estimated cold-start impact: under +1ms per scan on a 1k-file repo. The Hoeffding lookup is one indexed SQL select per scan call (not per file), amortised to ~0.

## Disqualifiers / why this might be wrong

1. **Cold-start dominates for casual users.** The headline win requires `n_k >= 30` history events per repo. On a fresh checkout the table is empty and V27 falls back to the existing `DEFAULT_IGNORE_DIRS`, contributing nothing. The set of repos where Redcon is used heavily *and* contains large untracked vendor/generated trees may be small. In that intersection the static `DEFAULT_IGNORE_DIRS` plus a one-line `redconignore` user file would deliver the same effect with zero history machinery. V27 only beats a static ignore when the user does not write the static ignore.
2. **Task-class drift may be wider than three buckets.** A user whose tasks split across "test", "docs", "perf", "deps", "schema migration", "i18n" finds that three classes shatter into noise (each `n_k` < 30) and the bound stays `None` for everyone. Refusing to prune is the safe failure mode but the latency win evaporates. We could enlarge the class taxonomy but then the Hoeffding sample-complexity bites: doubling classes halves `n_k` and *quadruples* the eps for the same confidence, which is the wrong scaling.
3. **The `fetched` event is hard to define faithfully.** A path that an agent merely *thought about* (e.g. saw it in a `redcon plan` audit and decided not to re-read) is indistinguishable from "never fetched". Without instrumentation in the agent itself we conflate "rejected with intention" with "never considered" and may over-aggressively prune kinds whose absence is the agent's own optimisation working correctly. This is a confounder that no SQL query can separate.
4. **Already partially shipped.** `redcon/scorers/history.py` *already* applies `history_ignored_file_penalty` per-path when the same path was ignored in similar prior runs. V27 is the **kind-aggregated** form of that mechanism. The marginal improvement over per-path history is exactly the cases where a path is novel (no prior individual history) but its kind is well-attested (sibling files in the same kind were never read). Useful, but not a new dimension - more like "history.py with a coarser key".
5. **Security/privacy adjacency.** Deriving a per-user `negative_space` table from a shared history database is a fingerprint of how the user works. It's local, but if the `.redcon/history.db` is committed by accident (the file is under `.redcon` which is in `DEFAULT_IGNORE_DIRS`, so this is unlikely but not impossible) the table leaks task vocabulary and reading habits. The mitigation is the same as today's `RUN_HISTORY_FILE` - documented, opt-in, never network-shipped. The risk doesn't grow but it *is* additive.
6. **Inverse-V26 redundancy.** If V26 (replay-value tracker) is built and gives a positive boost to re-read files, the relative ranking already handles "never-read" implicitly: those files float to the bottom and fall outside top-K. V27's hard-drop and scan-stage early-out add real latency, but the **token** win largely overlaps with V26. Building both yields diminishing returns on the token axis.

## Verdict

- Novelty: **low-medium**. The Hoeffding-bound framing is a clean rigorous wrapper, but at heart this is "auto-`.gitignore` from observation". `redcon/scorers/history.py` already does the per-path version of this story.
- Feasibility: **high**. The schema fits the existing SQLite history file with a single additive table + view; the integration points are surgical; cold-start failure mode is exactly current behaviour.
- Estimated speed of prototype: **3-5 days** for the SQLite table + classifier + Hoeffding bound + scan-stage early-out + tests; **+1 week** for MCP `fetched`-event instrumentation across `tool_compress`/`tool_search` and a stable A/B harness comparing scan latency with V27 on vs off across 5 sample repos.
- Recommend prototype: **conditional-on** running the synthetic-trajectory harness on at least three real repos with mixed vendor/generated trees first. If the median `f_neg` (fraction of candidates entering the never-read bucket) exceeds 0.20, build it. If `f_neg < 0.10` across the sample, V27 reduces to a stylish reformulation of the existing `DEFAULT_IGNORE_DIRS` and is not worth the database-schema complexity.
