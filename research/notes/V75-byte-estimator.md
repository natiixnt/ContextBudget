# V75: Tokenizer-free byte-level estimator with empirical calibration table

## Hypothesis

A purely byte-counting estimator, parameterised by a small per-character-class
or per-schema lookup table calibrated against `tiktoken cl100k_base` on the
Redcon fixture corpus, can replace the current `_tokens_lite.estimate_tokens`
heuristic (`ceil(len/4)`) with much lower error while remaining fully
tokenizer-free, dependency-free, and orders of magnitude faster than
`tiktoken.encode`. The claim: a 1-D bytes-per-token (B/tok) ratio per
*schema* (one number per compressor type) gets the median fixture under
3% rel error against tiktoken, and ~6-10% mean rel error - a 4-7x
improvement over the current lite heuristic. A per-character-class linear
model adds parameters but does **not** improve out-of-sample MAE on a
21-fixture corpus and overfits with only one to three fixtures per schema.
The right operating point is "per-schema 1-D table".

## Theoretical basis

The cl100k BPE tokenizer is a deterministic prefix-coder over bytes whose
output rate per byte is bounded between 1 token / max-merge-length and
1 token / 1 byte. For a stationary source X over UTF-8 bytes, the expected
token length is

    E[T(x)] = |x| * tau(p_X)

where `tau` is a source-dependent rate that depends only on the byte-class
distribution of X (because BPE merges depend only on local byte sequences),
not on x's length. For a finite mixture of k disjoint sources S_1..S_k
(diff lines, paths, code, prose), if a fixture is drawn purely from S_i then
its B/tok ratio is exactly `1/tau_i`, and a per-source ratio is the MMSE
estimator of `T(x)` given only `|x|` and the source label.

Back-of-envelope for the per-class linear model: for each character class j
let `omega_j = E[tokens contributed per char of class j]`. Modelled as a
linear additive contribution

    T_hat(x) = sum_j n_j(x) * omega_j         (1)

where `n_j(x)` is the count of class-j chars. Solve `omega` by
non-negative least squares against tiktoken truth. Equation (1) is exact
only if BPE merges respect class boundaries, which they don't:
`def foo(`  merges `def `+`foo` differently from `def bar(`, but both
contribute the same per-class counts. The residual is the merge-context
error, which equation (1) cannot capture. Empirically (see numbers below)
this residual is large enough that the linear model only matches the
per-schema 1-D ratio in-sample and is worse out of sample.

A second derivation - why per-schema BPT works at all. Redcon's command
output is generated by a small set of compressors that each emit
near-stationary text (diff hunks, ls listings, pytest fail blocks). The
Shannon-cross-entropy between schemas is high, but **within a schema**
the byte-class distribution and the local-Markov structure are stable
across fixture sizes (a 64 B status block and a 6 KiB status block have
roughly the same letter/punct/digit ratios). Thus the per-schema mean
B/tok is asymptotically efficient for that schema. Variance in B/tok
across same-schema fixtures (measured below: range 2.13 - 2.44 within
git_diff, 1.87 - 2.05 within ls) is the irreducible noise of the
estimator.

## Concrete proposal for Redcon

Two-tier extension to `redcon/cmd/_tokens_lite.py`. Default API stays
backward-compatible. New tier: `estimate_tokens(text, schema=None)`.

1. Bake a frozen calibration table (recomputed offline by
   `benchmarks/calibrate_lite.py`):

   ```python
   # redcon/cmd/_tokens_lite.py
   _BPT_BY_SCHEMA = {
       "git_diff": 2.43, "git_status": 2.79, "git_log": 2.84,
       "pytest": 4.08, "cargo_test": 3.36, "go_test": 2.74,
       "npm_test": 3.54,
       "grep": 2.71, "ls": 1.91, "tree": 2.59, "find": 2.26,
       "lint": 2.85, "docker": 3.17, "pkg_install": 2.53,
       "kubectl": 2.84,
   }
   _BPT_BY_DOMAIN = {                  # fallback if schema unknown
       "diff": 2.43, "paths": 2.14, "code": 2.98, "prose": 2.86,
   }
   _BPT_GLOBAL = 2.58
   ```

2. New API path (additive, current call signature unchanged):

   ```python
   def estimate_tokens(text: str, *, schema: str | None = None) -> int:
       if not text:
           return 0
       nbytes = len(text.encode("utf-8"))
       bpt = (
           _BPT_BY_SCHEMA.get(schema)
           or _BPT_BY_DOMAIN.get(_SCHEMA_TO_DOMAIN.get(schema, ""))
           or _BPT_GLOBAL
       )
       return max(1, round(nbytes / bpt))
   ```

3. Wire `compressor.schema` through to the call sites in
   `redcon/cmd/pipeline.py::compress_command` and the floor-vs-ratio code
   in `redcon/cmd/budget.py`. None of them currently pass a schema; the
   fallback path leaves their behaviour at "global mean".

4. New `benchmarks/calibrate_lite.py` (~120 LOC) that reads
   `tests.test_cmd_quality.CASES`, runs each compressor at COMPACT, calls
   `tiktoken.cl100k_base.encode` on raw + compact, and emits the table
   above as a Python dict literal printed to stdout. Run before each
   release; CI assertion that the table matches.

5. Regression test: a handful of golden fixtures with cached tiktoken
   counts. Assert `estimate_tokens(text, schema=s)` is within 10% of the
   stored count. Tightens over time as the corpus grows.

No production hot-path change beyond a one-byte UTF-8 encode and a dict
lookup. The cold-import cost is two dict literals.

## Estimated impact

- **Token-estimation accuracy** (vs tiktoken on the 21-fixture corpus,
  raw + compact, mean rel error):

  | Estimator | Mean rel% raw | Median rel% raw | Mean rel% compact | Median rel% compact |
  |---|---:|---:|---:|---:|
  | M0 lite (current `len/4`) | 31.91 | 31.23 | 33.51 | 37.01 |
  | M1d byte/tok per **domain** (in-sample) | 18.48 | 12.28 | 15.89 | 15.45 |
  | M1s byte/tok per **schema** (in-sample) | **5.88** | **1.94** | 10.55 | 7.14 |
  | M2 per-class linear, per-domain (in-sample) | 17.52 | 11.74 | 17.64 | 20.51 |
  | M3 per-class + letter-run4 feature (in-sample) | 17.65 | 11.87 | 17.94 | 20.51 |
  | M5 domain BPT + class residual correction (in-sample) | 21.20 | 16.50 | 27.22 | 25.71 |
  | M1s LOO (honest generalisation) | 14.31 | 11.43 | 16.42 | 14.94 |
  | M1d LOO | 22.07 | 17.95 | 18.35 | 18.75 |

  The 3% target is met *for the median fixture, in-sample, on raw inputs*
  by per-schema BPT. It is **not** met on mean rel error or on
  compact-output prediction; mean stays at ~6% raw, ~10% compact in-sample
  and ~14-16% under leave-one-out. The per-class linear model is no
  better and overfits. M5 (linear residual correction on top of M1d) is
  strictly worse: 31% LOO mean. Data sparsity (1-2 fixtures per schema)
  is the bottleneck.

  Hits-within-tolerance, M1s in-sample:
  - <=3% rel error: 12/21 raw, 5/21 compact
  - <=5%:           14/21 raw, 6/21 compact
  - <=10%:          17/21 raw, 13/21 compact
  - <=20%:          20/21 raw, 18/21 compact

- **Latency on git_diff_huge (32 KiB, N=500)**:

  | | ms/call | speedup vs tiktoken |
  |---|---:|---:|
  | tiktoken cl100k | 5.19 |   1x |
  | M0 lite (current) | 0.00019 | 26 800x |
  | M1 BPT (proposal) | 0.00064 |  8 100x |
  | M3 class+lr4 feature | 3.15 |     1.7x |

  The dominant cost is the UTF-8 encode (`text.encode("utf-8")` -> byte
  length). A pure-ASCII fast path (`len(text)` if string is ascii) cuts
  M1 to ~0.0001 ms.

- **Affects**: `redcon/cmd/_tokens_lite.py` (replaced internals),
  `redcon/cmd/pipeline.py` (passes schema to estimator), `redcon/cmd/
  budget.py` (uses schema where available; global fallback otherwise).
  No scorer change. No cache-key change. No tier-floor change.

## Implementation cost

- Lines of code: ~150 (table 30, function 20, calibration script 80,
  regression test 20).
- New runtime deps: **none**. tiktoken is needed only by the offline
  calibration script, not at runtime. Stdlib only on the hot path.
- Risks to determinism / robustness / must-preserve:
  - **None to determinism**: a frozen table + integer division. Trivially
    same-input-same-output.
  - **Calibration drift**: if the cl100k BPE merges shift in a future
    tiktoken release, table entries become stale. Mitigation: ship the
    calibration script as part of release prep; CI gate on
    `tiktoken.__version__`.
  - **Schema unknown**: falls back to domain or global. The estimator
    never errors; worst case it silently degrades to ~18% rel error
    instead of ~6%.
  - **Locality / size variance**: per-schema BPT averages a 64 B
    git_status with a 32 KiB git_diff. The B/tok ratio is stable enough
    across sizes that this isn't a problem (variance within schema
    is ~5%) but very small inputs (< 30 B) sometimes hit a 50% relative
    error from tokenizer-edge effects (header tokens dominate). The
    `max(1, round(...))` floor and the no-input zero make tiny inputs
    safe in practice.
  - **Compact-text mean rel error of 10% remains above 3%**. The
    estimator is *not* tight enough to be the canonical token counter;
    it is a faster estimator on a hot path that already accepts
    approximation. `redcon.core.tokens.estimate_tokens` (tiktoken-
    backed) remains the canonical surface.

## Disqualifiers / why this might be wrong

1. **3% target unmet at the mean**: the prompt asked for "within 3% of
   tiktoken on Redcon's output domain". Under the strictest reading
   (mean rel error across the corpus), no estimator achieves this on
   the 21-fixture corpus - even per-schema 1-D ratio. Median yes (1.94%
   raw), mean no (5.88% raw, 10.55% compact). The shortfall is
   structural: BPE merges are non-additive in byte-class counts, so any
   purely byte-counting model has an irreducible residual of ~5-10%
   mean rel error. Closing that gap requires a real BPE prefix tree
   (V31 / V32 territory) or a custom tokenizer (V99). Acknowledge:
   the contribution here is "5-7x better than M0, dirt cheap, schema-
   addressable" rather than "tokenizer-equivalent".

2. **Sparse calibration corpus**: 21 fixtures across 15 schemas means
   most schemas have a single fixture - i.e. the per-schema B/tok is a
   one-sample mean. Leave-one-out drops M1s mean rel error from 5.9% to
   14.3%, exposing this. A robust calibration needs >=3 fixtures per
   schema, which means growing `tests.test_cmd_quality.CASES`. Until
   then, the in-sample numbers above are optimistic.

3. **Already covered by `redcon.core.tokens`**: that module IS the
   canonical tokenizer. Two estimators (`_tokens_lite` and
   `redcon.core.tokens`) already exist precisely so the cmd hot path
   can avoid the tiktoken import. The contract in `_tokens_lite.py`
   says: "must equal redcon.core.tokens.estimate_tokens for any
   string". Replacing M0 with M1s breaks that contract unless
   `redcon.core.tokens.estimate_tokens` is *also* swapped, which is
   either a 5-line change (acceptable) or a behaviour break for any
   external caller (regression). Need to check the callers of
   `redcon.core.tokens.estimate_tokens` before rolling out.

4. **Per-class linear model is a dead end on this corpus**: V75's stated
   methodology was "build a per-character-class lookup table". This
   experiment did exactly that (M2/M3) and found it does not beat the
   simpler 1-D ratio. The per-character-class hypothesis is falsified
   on the 21-fixture corpus. The methodology delivers M1s as the
   actually-effective answer, not M2.

5. **Compact-text MAE remains weak**: most existing budget-allocation
   code (e.g. `redcon/cmd/budget.py::select_level`) uses estimated
   tokens of *raw* input (to decide compression tier). Raw-input MAE
   is the relevant quantity - here M1s shines (5.9% mean, 1.9%
   median). But the same code also estimates compressed-output size
   downstream, where mean rel error is 10.5%. Picking M1s over M0
   improves both, but the ratio of improvement is smaller on compact
   text because compact text is a more aggressively rewritten domain
   that the calibration corpus only thinly covers.

6. **Domain mapping is hand-coded**: `pytest -> code`, `git_diff -> diff`,
   `ls -> paths`. If a future compressor (say a JSON-log compressor,
   V65) is added and forgotten in the domain map, it falls back to the
   global mean (2.58 B/tok). Mitigation: the registry already knows
   each compressor's `schema` string; assign a default domain in the
   compressor class itself rather than in `_tokens_lite.py`.

Per-class linear model M2 (`raw -> truth`), in-sample:

- letter weights vary 0.06 - 0.58 tokens/char across domains.
- digit weight is `~1.2` tokens/char in code (3-6 char hex/decimal IDs
  often tokenise as 2-3 tokens), `~0.5` in paths (numeric path indices
  rarely punctuated).
- newline contributes 0.07 (diff) to 1.65 (paths) tokens/char - huge
  spread because cl100k merges `\n   ` differently in path lists vs
  diff hunks.
- The wide spread is exactly what would dilute a single global linear
  model and is why per-domain class weights help in-sample. They do
  not help out-of-sample because each domain has 2-9 fixtures, so the
  six-parameter regression overfits.

Speed (best of 500 calls on 32 KiB `git_diff_huge` raw):

| estimator | ms/call | speedup vs tiktoken |
|---|---:|---:|
| tiktoken cl100k_base | 5.19 | 1.0 |
| M0 lite (current `ceil(len/4)`) | 0.00019 | 26 800 |
| M1 byte / B-per-tok dict lookup | 0.00064 |  8 100 |
| M3 per-class + letter-run feature | 3.15 |     1.7 |

The bottleneck for M1 is the `text.encode("utf-8")` call (32 KiB ->
~0.4 us); the arithmetic is one float divide. M3's seven-feature pass
walks the string char-by-char and is dominated by Python interpreter
overhead, not arithmetic. A C/Rust port of M3 would close to ~10 us
range but the in-sample MAE wins are not worth the implementation
cost.

## Verdict

- **Novelty**: low. Per-schema bytes-per-token is the most natural
  refinement of the existing `len/4` heuristic; it is exactly what
  V04 already noted in its calibration table (mean 2.93 B/tok raw,
  2.65 compact). What is new here is (a) the explicit table baked
  into `_tokens_lite.py`, (b) the rigorous comparison against a per-
  character-class linear model (which loses), and (c) leave-one-out
  honest generalisation numbers.
- **Feasibility**: high. ~150 LOC, no new dep on the hot path,
  determinism preserved, `tiktoken` confined to offline calibration.
- **Estimated speed of prototype**: 4-6 hours - one afternoon for the
  calibration script, table, regression test, and pipeline plumbing.
- **Recommend prototype**: **conditional-on-X**, X = "we are willing
  to grow `tests.test_cmd_quality.CASES` to >=3 fixtures per schema
  before relying on the LOO numbers". As-is, in-sample
  `_tokens_lite.estimate_tokens` would improve from 32% mean rel
  error to ~6% raw / ~10% compact at zero hot-path cost. That is a
  4-7x estimator-quality improvement for ~150 LOC and zero new deps,
  and is a clear win for any caller that is currently swallowing the
  M0 error (e.g. tier selection in `select_level`, run.json estimated
  token columns). It is **not** a "breakthrough" by the BASELINE
  definition (no reduction-pp on COMPACT, no 20% cold-start cut, no
  new compounding compression dimension); it is a quality-of-life
  estimator upgrade that lands the lite path closer to tiktoken
  without paying the merge-table cost.
